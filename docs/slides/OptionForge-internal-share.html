项目想解决的不是“怎么再造一个策略仓库”，而是“怎样把策略研究、运行、观测和协作变成可以重复交付的工程动作”。

`application / domain / infrastructure` 明确边界，让策略逻辑不再挤成一团。

主题：期权策略脚手架的产品化方向 面向对象：策略研发 / 平台 / AI coding 协作
从零开始最贵的，往往不是代码量

为什么不是再写一个“策略模板”

02 / 12

多数策略仓库真正拖慢速度的，是工程层面的重复劳动：边界不清、入口分散、环境不稳定、观测缺位，以及协作时上下文失控。

问题 01

逻辑与基础设施糊在一起

合约选择、风控判断、监控落盘、网关对接常常挤在同一层里。

问题 02

入口越多，回归越难

回测、实盘、配置校验、监控各有脚本，经验无法沉淀成流程。

问题 03

新人和 agent 上手成本高

仓库一大，既不知道先看哪，也不清楚哪些文件能改、哪些别碰。

问题 04

策略能跑，不代表能交付

缺 monitor、缺部署、缺配置治理，最终只在作者机器上好用。

这个脚手架的回答

  • 用分层架构隔离流程编排、交易规则和外部依赖。
  • 把 `create / init / run / backtest / validate / doctor / focus` 收敛成统一 CLI。
  • 让 `.focus` 导航文件与 manifest 一起约束 agent 的可编辑边界。
  • 从第一天就把 Web monitor、事件快照和 Docker 部署放进项目骨架。
不是现成策略 是策略研发底座 强调复用与演进
核心目标：把高频重复工程从每个策略项目里抽离出来 结果：策略迭代聚焦在 alpha、风控与执行细节上
项目定位

它提供的不是收益曲线，而是可持续迭代的地基

03 / 12

仓库已经内置了策略项目最常见的四类成本中心：结构拆分、能力编排、AI 导航与运维观察。团队不必每次开新策略都重新把这些轮子装一遍。

A

分层结构

`StrategyEntry` 把 application、domain、infrastructure 串起来，保持调用路径清晰。

B

领域能力

selection、signal、pricing、combination、risk、execution、hedging 都有落点。

C

AI-first 导航

D

Web monitor、日志目录、PostgreSQL 与 Docker Compose 一起就位。

一句话总结：先把工程骨架搭稳，再把精力留给策略本身 仓库角色：期权策略研发的“产品化起点”
