# Python 问答式 CLI 创建器方案

## 1. 背景与结论

本轮结论是：CLI 主线不采用 Ink，也不单独引入 Node 子项目来承载创建器体验。

更合适的方向是继续使用 Python 实现“问答式创建器”，借鉴 `create-t3-app` 的产品体验，而不是照搬它的技术栈。原因如下：

- 当前仓库已经具备较完整的 Python scaffold 规则、preset 定义、能力依赖校验和文件生成逻辑
- 目标体验本质上是“低摩擦问答式创建器”，而不是“复杂终端 UI 应用”
- 继续复用现有 Python 逻辑，可以避免双栈维护和规则漂移
- 对 GitHub 新用户而言，重点是让 `option-scaffold` 更容易上手，而不是强调底层实现技术

因此，本方案聚焦于把现有 CLI 演进为一个更接近 `create-t3-app` 的 Python 问答式创建器，并且只覆盖 `create` 体验，不扩展到 Ink、Node worktree 实验或其他命令的交互式改造。

## 2. 核心体验目标

目标是让首次使用仓库的用户只需执行一次命令，就能通过交互完成期权策略工作区的 setup。

首版应达到以下体验：

- `option-scaffold` 无参数直接进入主菜单
- 主菜单默认高亮“创建策略工作区”
- 项目名提供默认值 `alpha_lab`，直接回车可通过
- preset 默认选择 `custom`
- 高级模块配置默认折叠，只有用户选择“我要自定义”时才展开
- 生成完成后默认自动执行安装、校验、git 初始化
- 对非交互场景保留现有 flags 路径，不强制进入问答模式

## 3. 主菜单与问答流程

首版交互流程固定如下：

### 第 1 步：显示主菜单

主菜单至少包含以下选项：

- 创建策略工作区
- 查看示例
- 环境诊断
- 退出

默认高亮“创建策略工作区”，保证新用户直接回车就能进入主路径。

### 第 2 步：输入项目名

- 提示用户输入项目名
- 默认值为 `alpha_lab`
- 用户直接回车时，沿用默认值

### 第 3 步：选择 preset

- 展示可用 preset 列表
- 默认值为 `custom`
- 首版不把 `ema-cross` 设为默认项

选择 `custom` 的原因是它约束最少、可解释性最好，也最适合作为通用起点。

### 第 4 步：是否自定义模块

- 询问用户是否“我要自定义”
- 默认值为“否”

如果用户不选择自定义，则直接采用当前 preset 的默认能力组合，不继续展示高级模块问题。

### 第 5 步：按需展开 capability 与 option

仅当用户选择“我要自定义”时，才进入高级模块配置：

- 先选择顶层 capability
- 再选择 capability 下的 option
- 如命中缺依赖或互斥关系，先展示自动修复预览，再让用户确认是否应用修复

该部分继续复用现有 Python 侧的规则和校验能力，不在 CLI 层重新定义业务语义。

### 第 6 步：目录冲突处理

如果目标目录已存在且非空，则提示用户选择冲突策略：

- `abort`
- `clear`
- `overwrite`

不再依赖用户预先记忆命令行 flags。

### 第 7 步：展示最终预览并确认

确认页至少展示：

- 项目名
- 目标目录
- preset
- 启用模块摘要
- 即将执行的后置动作

用户确认后，才进入实际生成阶段。

### 第 8 步：执行生成

执行 `create` 对应的 Python scaffold 流程，生成项目目录和配置文件。

### 第 9 步：自动执行后置动作

默认继续执行以下动作：

- 创建虚拟环境
- 安装依赖
- 执行 `validate`
- 初始化 git

### 第 10 步：输出 next steps

结束时输出清晰的后续命令建议，包括：

- `cd <project>`
- 如跳过安装，则输出安装命令
- 如跳过校验，则输出校验命令
- 给出最小 `option-scaffold run` 示例

## 4. 默认行为与实现细节

### 4.1 无参数默认进入主菜单

- `option-scaffold` 在交互终端中无参数执行时，不再直接显示 help
- 改为进入主菜单
- 非交互终端或显式 flags 场景下，继续保留现有命令行行为

### 4.2 默认项目名

- 默认项目名统一使用 `alpha_lab`
- 该默认值应在主菜单问答路径和相关默认创建路径中保持一致

### 4.3 默认 preset

- 默认 preset 选择 `custom`
- 保持首版体验稳定、通用、低假设

### 4.4 高级模块默认折叠

- 用户不主动选择“我要自定义”时，不展示 capability 和 option 问题
- 这样可以显著缩短首次创建路径，贴近 `create-t3-app` 的低摩擦体验

### 4.5 自动执行默认开启

后置动作默认值如下：

- `install = true`
- `validate = true`
- `git_init = true`

对应含义如下：

- `install`：自动创建虚拟环境并安装依赖
- `validate`：自动执行 `option-scaffold validate --config config/strategy_config.toml`
- `git_init`：自动初始化 git 仓库；如果目标目录已经位于 git 仓库中，则跳过初始化并明确提示

### 4.6 失败处理

若生成之后的任一步后置动作失败：

- 不回滚已生成文件
- 明确指出失败步骤
- 输出用户可手动执行的后续命令

## 5. 实现边界

本轮方案边界如下：

- 只记录文档方案，不扩展到 Ink、Node 子项目或 worktree 实验
- 首版只覆盖 `create` 体验，不把 `run`、`backtest`、`doctor` 一并改造成问答式
- 不新增 facade、coordinator 一类中间层
- CLI 层继续直接调用现有 scaffold 和相关服务逻辑
- 高级模块规则、preset、能力依赖与互斥关系继续复用 Python 现有实现

## 6. 推荐落地位置

建议的代码落点如下：

- 入口层：`src/cli/app.py`
- 创建器交互层：`src/main/scaffold/prompt.py`
- 创建编排层：`src/main/scaffold/project.py`

职责建议如下：

- `src/cli/app.py` 负责无参数进入主菜单和命令分发
- `src/main/scaffold/prompt.py` 演进为主问答入口，承载主菜单和创建向导
- `src/main/scaffold/project.py` 继续负责创建流程编排，避免把业务规则搬进 CLI 展示层

## 7. 验收标准

当以下结果全部满足时，可认为首版实现达标：

- `option-scaffold` 直接进入主菜单
- 连续接受默认值时，可创建默认项目 `alpha_lab`
- 不选“我要自定义”时，不会出现高级模块问题
- 默认创建完成后，会自动尝试安装、校验、初始化 git
- 目录已存在且非空时，会明确询问 `abort / clear / overwrite`
- 非交互场景下，现有 flags 仍然可用

## 8. 最小测试验收用例

### 8.1 交互默认流

- 在交互终端执行 `option-scaffold`
- 直接回车进入“创建策略工作区”
- 项目名使用默认值 `alpha_lab`
- preset 使用默认值 `custom`
- 不开启“我要自定义”
- 成功生成项目并触发默认后置动作

### 8.2 自定义流

- 在向导中选择“我要自定义”
- 成功进入 capability 与 option 选择
- 能完成模块定制并通过规则校验

### 8.3 冲突流

- 当目标目录非空时，向导必须展示 `clear` 和 `overwrite` 等策略选择
- 用户选择不同策略后，行为与预期一致

### 8.4 自动执行流

- 创建完成后，默认执行虚拟环境、依赖安装、`validate`、git 初始化

### 8.5 失败流

- 若安装或校验失败，已生成目录保留
- 终端输出失败步骤和 next steps

### 8.6 非交互流

- `--no-interactive` 及现有 flags 继续可用
- 非交互终端不强制进入问答式流程

## 9. 最终结论

本项目下一阶段的 CLI 演进主线，应当是“Python 实现的问答式创建器”。

重点不是把 CLI 做成终端前端应用，而是把 `option-scaffold` 做成一个新用户几乎不需要学习命令参数、只需一次启动就能完成策略工作区 setup 的创建器。这个方向比 Ink 更符合当前项目基础，也更适合作为提升 GitHub 转化和 star 吸引力的主线方案。
